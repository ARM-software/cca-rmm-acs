
# Arm RMM ACS Device Assignment Testcase checklist
-----------------------------------------------------

This document presents the mapping of the rules in the RMM specification to the
test cases and the steps followed in the tests. This also provides the information
about the test coverage scenarios that are implemented in the current release of
ACS and the scenarios that are planned for the future releases.

| Test Number | Test Name | Test Assertions | Test Approach | Validated by ACS |
| --- | --- | --- | --- | --- |
| 1 | da_offchip_pcie | Device Assignment for offchip pci device | 1.Host: PCIe enumeration and device discovery<br>- Boot in NS Host.<br>- Initialize PCIe and perform enumeration.<br>- Read RMI_FEATURES and confirm DA support.<br>- Find a PCIe endpoint with DOE capability.<br>- Identify the root port and discover BAR/MMIO ranges for the selected device.<br><br>2.Host: Create PDEV and SPDM session establishment<br>- Allocate/delegate granules for PDEV and its aux objects.<br>- Populate PDEV parameters.<br>- Call RMI_PDEV_CREATE. Call RMI_PDEV_GET_STATE and verify PDEV_NEW.<br>- Call RMI_PDEV_COMMUNICATE until PDEV states is PDEV_NEEDS_KEY. Cache the certificate chain.<br>- Extract the device public key from the cached certificate chain.<br>- Call RMI_PDEV_SET_PUBKEY. Call RMI_PDEV_GET_STATE and verify PDEV state is  PDEV_HAS_KEY.<br>- Continue RMI_PDEV_COMMUNICATE until PDEV state reaches PDEV_READY.<br><br>3.Host: Create Realm and VDEV<br>- Create a Realm with DA enabled.<br>- Allocate/delegate granules for VDEV and its aux objects.<br>- Call RMI_VDEV_CREATE. Call RMI_VDEV_GET_STATE and verify VDEV_NEW.<br>- Call RMI_VDEV_COMMUNICATE until VDEV state reaches VDEV_UNLOCKED.<br><br>4.Host: Collect device attestation evidence<br>- Call RMI_VDEV_GET_MEASUREMENTS. Complete required RMI_VDEV_COMMUNICATE and keep state VDEV_UNLOCKED. Cache the measurements.<br>- Call RMI_VDEV_LOCK. Complete RMI_VDEV_COMMUNICATE and verify VDEV_LOCKED.<br>- Call RMI_VDEV_GET_INTERFACE_REPORT. Complete required RMI_VDEV_COMMUNICATE and keep state VDEV_LOCKED. Cache the interface report.<br>- Call RMI_VDEV_START. Complete RMI_VDEV_COMMUNICATE and verify VDEV_STARTED.<br><br> 5.Realm: verify device information<br>- Realm boots and checks RSI_FEATURES for DA support.<br>- Host passes vdev_id to Realm.<br>- Realm calls RSI_VDEV_GET_INFO.<br>- Realm verifies digests from RMM against Host-cached:<br>&nbsp;&nbsp;- VCA<br>&nbsp;&nbsp;- certificate<br>&nbsp;&nbsp;- measurements<br>&nbsp;&nbsp;- interface report<br>- Realm decodes the interface report and extracts MMIO ranges.<br><br>6.Host/Realm: Validate mappings<br>- Realm requests mapping validation for each MMIO range using RSI_VDEV_VALIDATE_MAPPING.<br>- Host observes RMI_EXIT_VDEV_MAP exits.<br>- For each exit, Host:<br>&nbsp;&nbsp;- validates REC exit fields<br>&nbsp;&nbsp;- maps device memory<br>&nbsp;&nbsp;- calls RMI_VDEV_VALIDATE_MAPPING<br>&nbsp;&nbsp;- checks RTT entries become HIPAS=ASSIGNED_DEV and RIPAS=DEV<br>- Realm confirms the mappings are valid for the reported IPA ranges. <br><br>7. DMA state of a VDEV is VDEV_DMA_DISABLED so when realm performs DMA using the test engine from source buffer to destination buffer in Realm memory, the SMMU blocks traffic from the device to the realm's protected IPA space.  <br><br>8.Realm: Enable DMA and realm verify data movement<br>- Realm calls RSI_VDEV_DMA_ENABLE.<br>- Realm performs DMA using the test engine from source buffer to destination buffer in Realm memory.<br>- Realm verifies DMA success by comparing source and destination buffers.<br><br>9.Teardown<br>- Realm calls RSI_VDEV_DMA_DISABLE.<br>- Host unmaps device memory using RMI_VDEV_UNMAP and verifies RTT entries become HIPAS=UNASSIGNED, RIPAS=DESTROYED.<br>- Host tears down VDEV:<br>&nbsp;&nbsp;- RMI_VDEV_UNLOCK<br>&nbsp;&nbsp;- RMI_VDEV_DESTROY<br>- Host tears down PDEV:<br>&nbsp;&nbsp;- RMI_PDEV_DESTROY | Yes |
